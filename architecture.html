Earnings Extractor — Architecture

A deterministic pipeline with constrained LLM step(s). Click any stage to expand. Your code controls the order — the model never does.

30-sec version: One file reads the PDF, one classifies it, one makes the LLM call to pull numbers with quotes — then several files do the reliable deterministic work (normalize units, validate, flag for review), a human approves, and one file writes the Excel. The order is fixed in pipeline.py. That's why it's a pipeline, not an agent.

Per-document pipeline · pipeline.py → process_single_pdf() (fixed order)
1
Ingest the PDF
ingest.py · read_pdf_pages, read_pdf_metadata
I/O▶
Turns the PDF into per-page text + metadata so everything downstream works on text, not the raw file.
if asked, say"Reads the PDF into page-by-page text — the input layer for the whole pipeline."
▼
2
Classify & select pages
classify.py · classify_document, select_extraction_pages
Deterministic▶
Decides report vs earnings-call transcript by structure (not keywords) and picks only the pages worth reading, so the model isn't fed the whole filing.
why it matters"This is what lets it handle both Tesla's tables and Citi's prose — and it controls cost by not sending every page to the model."
▼
3
LLM extraction — the only model read
extractor.py · extract_metrics_live_with_usage | recorded.py · extract_metrics_recorded
LLM▶
One structured call returns each metric as {value, source_page, source_quote, confidence}. This is the only place the model reads the document. recorded.py replays saved responses so it runs offline with no API key.
if asked, say"The model does only the fuzzy reading — find the number and quote its source. It never decides control flow."
▼
4
Complete template rows
validation.py · complete_template_rows
Deterministic▶
Ensures every required client-template field exists as a row — missing ones become blank + review-flagged, never guessed.
▼
5
Resolve company identity
identity.py · resolve_company_identity, apply_company_identity
Deterministic▶
Fills company name / ticker from text + metadata when the filing implies rather than labels it — a thing the model routinely leaves blank.
▼
6
Capital-return (buybacks & dividends)
validation.enrich_capital_return_text · capital_return.resolve_capital_return_narrative
DeterministicLLMLIVE▶
Builds the combined "Buybacks and dividends" field from the source text. Deterministic first; only if the cell is weak does a live model call phrase it — and the number is checked against the source. Stays review-flagged.
note"This is one of the extra live calls — part of why live mode isn't a single call."
▼
7
Normalize units & scale
validation.repair_table_scale · normalize.py · normalize_metrics, normalize_currency_to_usd_millions
Deterministic▶
Converts everything to one canonical form: USD millions, gross margin as percentage points, EPS as a plain number. Uses the quote's explicit amount to avoid double-scaling.
the money line"This turns Citi's '$21.6 billion' into the right number — a 1000× error the model can't be trusted to get. It's the biggest reason the deterministic layer takes raw 15/18 → 18/18."
▼
8
Line-item correction
line_item_selector.py · apply_line_item_selection
LLMLIVE▶
If the model grabbed a line adjacent in meaning (a revenue subtotal instead of total revenue), this re-selects the definition-matching line from the cited page. Caged so the value stays grounded.
note"A separate live model call — the generalizing fix for wrong-line picks."
▼
9
Repair source pages
validation.py · repair_source_pages
DeterministicLIVE▶
Live extractions can mis-cite a page; this snaps each quote to the page that actually contains it before the citation validator runs.
▼
10
Validate — consistency, magnitude, review flags
validation.py · validate_metrics, check_free_cash_flow, check_value_magnitudes
Deterministic▶
The reliability gate: quote must exist on the cited page, the number must appear in its quote, consistency holds (OCF − capex = FCF), magnitudes are sane (net income ≤ revenue). Anything off gets needs_review = true with a plain-English reason.
if asked, say"Validation never edits a value — it only flags it for a human, with the reason."
▼
11
LLM verifier (safety net)
verifier.py · verify_template_metrics
LLMLIVE▶
On live runs, the model re-checks each populated value against its quote and flags semantic errors deterministic rules can't enumerate (wrong line item, full-year vs the quarter). It only adds flags — never edits a value.
note"The 3rd/4th live call. Add-flags-only keeps it safe; recorded mode skips it to stay byte-for-byte reproducible."
▼
12
Assemble draft
pipeline.py · build_draft_run → draft_metrics.json
I/O▶
Writes the draft run: normalized rows, evidence, validation flags, and token usage. Inputs to review — not the final client Excel.
▼
13
Human citation review
review.py · build_review_items, write_review_artifacts, load_review_decisions
Human gate▶
Shows every value with its page, quote, confidence, and flag reason, and records approve / reject / note. (The web UI is the nicer version of this surface.)
the headline"Nothing reaches the client sheet until a human approves the citation behind it."
▼
14
Export the workbook
export.py · export_reviewed_run, map_metric_to_client_cell, format_*
I/O▶
Writes extractions.xlsx: client-template first sheet + Metrics / Review / Evidence tabs, formatting each cell ($<n>B strings, margin as decimal). Only approved values land in the client sheet; unsupported cells stay blank.
How a user upload actually enters the pipeline · api/extract.py · api/process.py
The web path: the browser (app/page.tsx) POSTs the PDF as multipart/form-data. The serverless handler does NOT re-implement anything — it unpacks the file, writes it to a temp .pdf, and hands off to the exact same pipeline.extract() the CLI uses. One engine, two front doors (CLI + web).
A
Receive & unpack the upload
api/extract.py · do_POST → _read_request → _read_multipart
I/O▶
Parses the multipart body and pulls out {filename, file_bytes} (the raw PDF bytes). Recorded/demo mode instead picks a bundled golden PDF (tesla/citi) — no upload needed.
nuance"api/process.py is the same but also builds the Excel; api/extract.py returns metrics-only for the preview."
▼
B
Write bytes to a temp PDF
api/extract.py · _process → _write_input_pdf
I/O▶
Sanitizes the filename, writes file_bytes into a TemporaryDirectory as a real .pdf, and returns a pdf_path. From here on it's identical to a CLI run on a local file.
▼
C
Hand off to the shared engine
pipeline.extract() → process_single_pdf() → (stages 1–14 below)
Same pipeline▶
extract() finds the PDF(s), runs process_single_pdf (the fixed 14-stage sequence above), and writes draft_metrics.json. The web handler then reads that draft back and — in process.py — auto-approves & calls export_reviewed_run to return a base64 Excel.
the point"The web layer is a thin adapter. All the logic lives in the one pipeline — that's why CLI and web can't drift apart."
What the data actually looks like at each hop (the shapes)
Clears up the #1 confusion: ingest reads every page → classify runs on all pages but its verdict is overridden to "earnings_report" (audit-only) → select_extraction_pages picks ≤6 pages → the model only ever sees those 6 pages, not classify's output. After the one model call, every later stage mutates the same list[MetricRow] in place.
Hop Function In → Out (data shape)
Upload _read_multipart HTTP body → {filename, file_bytes} → temp pdf_path
1 · ingest read_pdf_pages pdf_path → list[PageText]  (ALL pages: page_number, text, char_count)
2a · classify classify_document list[PageText] → DocumentClassification  ⚠ verdict overridden to "earnings_report"
2b · select select_extraction_pages list[PageText] → ≤6 PageText  (page 1 + highest _page_relevance)
3 · extract LLM extract_metrics_live_with_usage ≤6 PageText → MetricsBatch = list[MetricRow]  {value, source_page, source_quote, confidence}
4–11 complete / identity / normalize / validate … same list[MetricRow] mutated in place — units fixed, rows completed, needs_review set
12 · assemble build_draft_run list[MetricRow] → DraftRun → draft_metrics.json
13–14 review → export_reviewed_run DraftRun + decisions → extractions.xlsx  (only approved cells)
Cross-cutting (not stages — they support every step)

schema.py — data contracts

Typed shapes (Pydantic) for metric rows & draft runs. repair_metric_batch rejects/repairs malformed model JSON so bad output never flows downstream.

recorded.py — offline mode

Replays committed model responses from recorded_responses/ so reviewers (and the eval) run with no API key, deterministically.

evaluation/ — the gate

golden_metrics, scoring, tolerances. Golden values live only here; a test blocks the extractor from importing them. This is the 90% gate.

Two entry points, one engine

batch = folder → one workbook, skips bad files, stamped DRAFT/UNREVIEWED. extract→review→export = the rigorous human-gated path. Both call process_single_pdf.

Live mode = ~3–4 model calls / PDF: extraction (3) + line-item (8) + verifier (11) + sometimes capital-return (6). Everything else is deterministic.  ·  Recorded eval: Tesla 9/9, Citi 9/9  ·  deterministic layer: raw 15/18 → 18/18  ·  cost: a few cents/PDF, ~100×+ cheaper than analysts.

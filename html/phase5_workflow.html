PHASE 5 · THE HUMAN REVIEW GATE

Earnings Extractor — Phase 5, explained

Phase 4 produced a draft you can trust and reproduce — every number canonical, every quote fact-checked, all nine client fields present, runnable offline. But a draft is still a machine's first guess. Phase 5 puts a human in the loop. It turns draft_metrics.json into a review package: a queue of every value, a plain-Markdown evidence report, and a clickable review.html where a person approves, rejects, or flags each number — producing a review_decisions.json that the Phase 6 export will require before it writes a final Excel. Nothing reaches the client workbook without passing through this gate.

Theme: draft → reviewed, sign-off-ready result New command: review New: queue · evidence report · review UI · decisions Next → Phase 6: final Excel export

If you read nothing else

The review command reads a finished draft run and writes four artifacts next to it: review_queue.json (every metric, machine-readable), evidence_report.md (a readable audit table), review.html (the interactive review page a human clicks through), and optionally a demo review_decisions.json. The whole point: every value — even high-confidence ones — gets a visible citation review before it can become a final client number. Phase 5 only collects decisions; it deliberately does not export Excel (that's Phase 6) and does not touch extraction.

All the heavy lifting lives in one new file, earnings_extractor/review.py, wired to the CLI as review. It reuses the same DraftRun schema and review flags Phase 4 produced — it never re-derives them.

The four decision choices are exactly: approved, rejected, needs_fix, not_applicable. A blank/unsupported field (e.g. operating income on a bank transcript) defaults to not_applicable; an attention-flagged value defaults to needs_fix; everything else to approved — but these are defaults a human is expected to confirm, not final answers.

Two ideas that keep the gate honest

🚩 requires_attention

Phase 5 does not reinvent "is this risky?" logic. It reuses Phase 4's needs_review flag and only adds one new trigger — source_resolution_failed (it couldn't tell which PDF a metric came from). So: requires_attention = needs_review OR source_resolution_failed. One source of truth, no drift.

🗂️ Which PDF did this come from?

Draft metrics don't yet carry their own source file, so Phase 5 resolves it with a heuristic: one doc → use it; otherwise a unique document type; otherwise match company/ticker to the filename; otherwise "unknown" + flag for attention. A known v1 limitation, logged in DECISIONS.md.

Demo decisions vs. real human decisions

Both are valid review_decisions.json files; the is_demo flag is how Phase 6 tells them apart.

Demo (CLI --demo-decisions)

queue→ auto rules→ is_demo: true

Deterministic, for verification only. Blank → not_applicable, placeholder/unresolved evidence → needs_fix, populated + evidence → approved. Each row carries a note saying a human must still confirm.

Human (browser download)

review.html→ your clicks→ is_demo: false

The downloaded file carries the run_id, created_at, reviewer name, and each decision — and validates through load_review_decisions(), the same gate Phase 6 uses.

Honesty caveat (carried into Phase 6): the download pre-fills untouched items with those same defaults. So clicking Download without reviewing yields a "human" file that auto-approved high-confidence values. Phase 6's export gate therefore treats "reviewed" as resolved attention state, not merely "a decision row exists" — an attention-flagged field left at bare approved won't export without a visibly-marked override.

The decision-file validator

load_review_decisions() is the guard Phase 6 will lean on. It refuses anything malformed:

RejectedWhy
invalid JSON / missing run_idNot a usable decisions file.
duplicate metric_idTwo conflicting decisions for one value.
unknown status / "pending"Only the four real decisions count; pending is not a decision.
mismatched run_idDecisions from a different extraction run can't be trusted against this draft.
demo & human filesBoth shapes are accepted as long as they're well-formed.
Phase 5 stops at structural validity. It does not enforce "every metric has a decision" — that completeness check belongs to Phase 6's export gate.

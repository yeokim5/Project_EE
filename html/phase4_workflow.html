PHASE 4 · TRUST, NORMALIZE & RUN OFFLINE

Earnings Extractor — Phase 4, explained

Phase 3 proved the machine could read a PDF and pull out metrics. But it only worked well on a clean Tesla report (9/9), and it needed an API key to run at all. Phase 4 is the trust-and-reliability layer. It makes every number consistent, checks every claim against its own evidence, guarantees the output always has all nine client fields, deterministically nails the company name, and lets a reviewer run the whole thing with no API key. This page walks the whole project as it stands now, and shows exactly what Phase 4 bolted on.

If you read nothing else

The project is a review-first earnings extractor. You give it an earnings PDF; it returns a draft of nine financial metrics, and every value is tied to the exact quote and page it came from. It never blindly trusts itself — anything weak or missing is flagged for a human. There are two walled-off worlds: the earnings_extractor/ package that does the work, and the evaluation/ package that grades it against a hand-verified answer key. The extractor is forbidden from importing the answer key (a test enforces this), so a good score can only come from actually reading the document.

Phase 4's job was everything between "the model returned some metrics" and "a human can trust the draft": clean up the numbers, fill in the template, confirm the company, fact-check each quote, and make it all runnable offline.

The five things Phase 4 added

🎞️ Offline mode

recorded.py — replays a saved model response from a JSON "cassette" so a reviewer can run the pipeline with no OpenAI key. --mode recorded.

🧮 Normalization

normalize.py — forces every number into one canonical unit. $21.6B, "4.1 billion", and 13400 all become millions of USD.

🆔 Company identity

identity.py — works out the company name deterministically (no AI) from titles, PDF metadata and filename, and demands on-page proof.

📋 Template completion

complete_template_rows() — guarantees all nine client fields exist as rows, inserting blank, review-flagged placeholders for any the model missed.

✅ Consistency checks

validate_metrics() — flags low confidence, checks each quote really appears on its cited page, and reconciles cash-flow math.

Carried over from Phase 3

PDF ingest, no-AI document classification, page selection, structured LLM extraction with evidence, schema validation — all still here, now feeding the new layer.

Resolve the company — deterministically

The company name isn't left to the model's guess. A no-AI resolver tries, in order: a transcript title ("Citi First Quarter 2025 Earnings Call"), then PDF metadata, then the filename — and each candidate must be found verbatim on a real page to count as confident. Tesla resolves to page 3 with confidence; anything only inferred from metadata stays review-flagged.

Normalize every number to one unit

Currency fields → USD millions (so $21.6B, "4.1 billion", and 13400 become 21600, 4100, 13400). Margins → percentage points. EPS → USD/share. Parentheses become negatives; capex is stored as a positive magnitude. If a value can't be parsed, it's flagged rather than mangled.

normalize.py → normalize_metrics()
in: messy values + unitsout: canonical numbers

PH4

Consistency checks — fact-check the model

Three deterministic guards: (a) any metric below 0.75 confidence is flagged; (b) every populated value's source quote must actually appear on the page it cites — if it doesn't, that's a hallucination signal and the row is flagged; (c) a cash-flow reconciliation: operating cash flow − capex should equal free cash flow within tolerance, else all three are flagged.

validation.py → validate_metrics(), check_free_cash_flow()
in: normalized rows + page textout: rows with honest review flags

Assemble & save the draft

Everything is packed into a DraftRun audit record — run id, UTC timestamp, mode, model, pages chosen, classification, and all metric rows — written to disk as JSON. inspect can then summarize it ("With evidence: 8/9", "Needs review: 3/9").

pipeline.py → outputs/<run>/draft_metrics.json
out: draft_metrics.json
How to read this: stages 1–4 are about getting a draft (Phase 3's territory, plus the recorded-mode switch). Stages 5–8 are about trusting that draft — they are the heart of Phase 4. The model gives a first guess; the deterministic layer cleans it, completes it, and fact-checks it before any human sees it.

Offline mode, a little deeper (the "cassette" idea)

Borrowed from how tests record HTTP calls and replay them. The point: a reviewer should be able to reproduce the exact pipeline output without paying for, or holding a key to, the OpenAI API.

live mode

PDF→ OpenAI call→ MetricsBatch

Needs OPENAI_API_KEY. Real model, real cost, fresh answer each run.

recorded mode

PDF→ read JSON cassette→ MetricsBatch

No key, no network. The saved response is loaded and run through the same repair + Phase-4 layers. Missing cassette → FileNotFoundError, never a fake.

Cassettes on disk: recorded_responses/TSLA-Q2-2025-Update.pdf.json, recorded_responses/citi_earnings_q12025.pdf.json

Why these five pieces, specifically

Phase 3 ended with an honest weakness: the Tesla report scored 9/9 but the Citi transcript scored 4/9, because numbers in spoken sentences ("on $21.6 billion of revenues…") are easy to misread, and the company name / units were inconsistent. Phase 4 attacks the reliability side of that gap rather than just tuning the prompt:

Phase 4 pieceProblem it fixes
normalize.py"$21.6B" vs "21600" vs "21.6 billion" — same fact, three forms. Grading and export need one canonical number.
identity.pyCompany name was guessed inconsistently. Now resolved deterministically and only trusted when proven on a page.
complete_template_rowsA missing field used to vanish. Now every client field always exists, blank-but-flagged if unsupported.
validate_metricsA confident-looking number with a quote that isn't actually on the cited page is a hallucination. Now caught.
recorded.pyThe required deliverable is a keyless, reproducible run for reviewers. Cassettes make the pipeline deterministic.
The unifying rule: the deterministic layer can flag, complete, and reformat — but it never invents a value. Blank stays blank (and flagged); a number only changes form, never substance. That's what keeps the "review-first" promise honest.

What one metric row carries through all this

Every row is a MetricRow (in schema.py). Phase 4 touches the value/unit/scale (normalize), the source fields (identity, quote check), and the review fields (completion + checks) — but the contract stays the same: a value is only worth anything if it carries its evidence.

How Phase 4 is proven

tests/test_phase4_processing.py locks in the new behavior so it can't silently regress. A sample of what it asserts:

Reproducibility & shape

Recorded Tesla emits exactly the nine template rows; an un-cassetted PDF in recorded mode fails clearly; completion inserts schema-safe blank placeholders.

Numbers & evidence

Billions/millions normalize correctly; margins, EPS and capex normalize; a fabricated quote not on its cited page gets flagged; cash-flow reconciliation passes and fails as expected.

The original guards still hold too: the no-cheat wall (test_no_cheat_imports.py) still fails the build if the extractor imports the answer key, so a high score can't come from peeking.

Earnings Extractor — how it works

A review-first tool that reads an earnings PDF and pulls out key financial metrics — but never just trusts itself. Every number it reports is tied to the exact quote it came from, and anything uncertain is flagged for a human. This page walks the whole machine, stage by stage, with the real files and functions behind each step.

Verdict: Phase 3 passed → cleared for Phase 4 Tesla report: 9/9 (100%) Citi transcript: 4/9 (44%) · Phase-4 work 29 tests pass · lint clean

The one idea behind everything

There are deliberately two separate worlds in this project that are not allowed to talk to each other:

🏭 The extractor — does the work

The package earnings_extractor/. It reads the PDF and produces draft metrics. It has no idea what the "right" answers are — it must actually read the document.

📋 The evaluation — grades the work

The package evaluation/. It holds the hand-verified answer key and scores the draft against it — like a teacher with a marking scheme.

The "no-cheat" wall: the extractor is forbidden from importing anything in evaluation/. A test (tests/test_no_cheat_imports.py) walks the code's syntax tree and fails the build if it finds even one such import. Otherwise the tool could "score 100%" by peeking at the answers instead of reading the PDF — making the score meaningless.

The pipeline — one PDF, six stages

This is what runs when you type python3 -m earnings_extractor extract <pdf> --out … --mode live. Orchestrated by extract() in pipeline.py.

⌘

Command line entry

You pick a subcommand: extract, inspect, or eval. The parser routes the call and catches errors cleanly.

cli.py → build_parser(), main()
1

Read the PDF → page text

Opens the PDF and pulls plain text out of every page, keeping the real 1-based page numbers so evidence can point back accurately. Built on the pdfplumber library.

ingest.py → read_pdf_pages() · returns list[PageText]
in: PDF fileout: page text + page numbers
2

Classify the document

Is this a formal earnings report (tables of numbers) or an earnings call transcript (people speaking)? Decided without any AI — purely from generic signals: how dense with numbers a page is, standard accounting phrases ("$ in millions", "unaudited"), and a speaker-turn pattern (Name:). If neither, it's marked unknown and skipped.

classify.py → classify_document(), classify_page()
earnings_reportearnings_call_transcriptunknown → skipped

These heuristics were deliberately rewritten to be generic — no company names, no hardcoded quarters — so the tool generalizes to PDFs it has never seen.

3

Pick the pages worth reading

A 30-page PDF is mostly filler. This ranks pages by relevance — numeric_density + 2 × marker_hits (markers = financial terms like "total revenue", "operating income", "EPS") — and keeps the top pages. Page 1 is always included for company identity. This is why Tesla's metric-rich page 4 always survives the cut.

classify.py → select_extraction_pages() (default max 6 pages)
in: all pagesout: ~6 highest-signal pages
4

Ask the AI to extract — with evidence

The selected page text is sent to OpenAI (gpt-5.4-mini, reasoning_effort=low) using Structured Outputs — the model is forced to return data in our exact schema, not free text. The prompt demands a source quote for every metric and says: if a value isn't supported, return null + needs_review rather than guessing.

extractor.py → extract_metrics_live() · client.responses.parse(text_format=MetricsBatch)
in: page text + doc typeout: MetricsBatch (raw model output)
5

Validate & repair the output

The model's response is run through strict Pydantic validation (extra="forbid", confidence must be 0–1, metric name and source quote can't be empty). Harmless wrapping (e.g. a stray {"data": …}) is unwrapped and enum casing fixed — but values are never filled in. Malformed output is rejected, not trusted.

schema.py → repair_metric_batch(), MetricRow validators
in: raw model outputout: validated MetricRow[]
6

Assemble & save the draft

Everything is packed into a DraftRun — a full audit record: a unique run id, UTC timestamp, model used, which pages were chosen, the classification, and all metric rows — then written to disk as JSON.

pipeline.py → builds DraftRun → outputs/<run>/draft_metrics.json
out: draft_metrics.json

What a single extracted metric looks like

Every row is a MetricRow (defined in schema.py). The fields below are why the tool can be "reviewed" rather than blindly trusted — note source_page, source_quote, needs_review and review_status.

metric_name — required, non-empty
value — number, text, or null
unit / scale — e.g. USD, millions
period / fiscal_period
gaap_or_non_gaap
segment — business segment if any
year_over_year_change
source_page — required
source_quote — required, non-empty
confidence — 0.0 to 1.0
needs_review — default true
review_reason — why flagged
review_status — pending / approved / …
company / ticker
The template the tool targets has 9 client fields: Company Name, Quarter, Total revenue, Earnings per share, Net income, Operating income, Gross margin, Operating expenses, and Buybacks & dividends.

How the score is computed

Separate from extraction. The evaluation/ package holds the hand-verified answer key and grades the draft field-by-field — within sensible tolerances, since "1,172" and "1172.0" should both count as correct.

The answer key

golden_metrics.py stores each correct value with the exact quote and page it came from. Some fields are intentionally marked "expected blank" — the tool earns points for not inventing a value that isn't in the document.

The grader

scoring.py → score_draft() matches each expected field to a draft metric by name, then checks the value within tolerance. _eval_bridge.py is the only sanctioned crossing point, via a dynamic import.

Value typeHow it's judged
currency_usd_millionsnumeric, within a scaled currency tolerance
epsnumeric, within ±0.01
percentage_pointsnumeric, within ±0.1 pp
text / free_textnormalized text match
expected_blank_reviewpasses only if value is empty and flagged for review
The grader even understands "$4.1 billion" → 4,100 (millions): _coerce_number() strips $, commas, and a trailing B/M/% before comparing.

Why Citi is harder: in a report, numbers sit in clean labeled tables. In a transcript, figures are buried in spoken sentences ("on $21.6 billion of revenues…"), so the model grabs the wrong line. This is a tool limitation, not an answer-key problem — every Citi golden value was verified to appear verbatim in the PDF. Fixing transcript extraction is the headline job of Phase 4.

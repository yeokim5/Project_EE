PHASE 6 · THE FINAL EXPORT GATE

Earnings Extractor — Phase 6, explained

Phase 5 collected a human's decisions into review_decisions.json but deliberately wrote no Excel. Phase 6 is the gate that turns reviewed decisions into the deliverable. The new export command reads a draft run plus its decisions, refuses to ship anything that wasn't properly reviewed, and — only when the gate is satisfied — writes the client-template workbook alongside a JSON sidecar and a Markdown audit report. The guiding rule: a number reaches the client sheet only after a human resolved it; otherwise the file is loudly stamped DRAFT.

If you read nothing else

The export command does three things in order: (1) rebuilds the review items from the draft and loads the decisions, (2) runs the review gate — and either blocks, or marks the run DRAFT, or lets it through as FINAL — and (3) writes the workbook: a first sheet that exactly matches the client's EarningsSample (1).xlsx template, plus three audit tabs (Metrics, Review Decisions, Evidence). It never re-extracts and never imports the eval fixtures.

All the logic lives in one new file, earnings_extractor/export.py, wired to the CLI as export. It reuses Phase 5's build_review_items() and load_review_decisions() — the same gate the reviewer's file already passed.

Try it yourself — the three export outcomes

From the project root. Recorded mode needs no API key. Steps 1–2 just rebuild a reviewed run; steps 3–5 are the three things export can do.

# 1 · build a draft from BOTH sample PDFs, then make the review package + a DEMO decisions file
.venv/bin/python -m earnings_extractor extract assesment_info --out outputs/run_001 --mode recorded
.venv/bin/python -m earnings_extractor review outputs/run_001 --demo-decisions outputs/run_001/review_decisions.json

# 2 · OUTCOME A — demo decisions are not real review, so a FINAL export is refused
.venv/bin/python -m earnings_extractor export outputs/run_001 --decisions outputs/run_001/review_decisions.json --out outputs/extractions.xlsx

What you should see: step 2 fails with a non-zero exit and a clear reason. Step 3 writes three files; the workbook's first tab is named Client Template DRAFT with an amber tab colour and a comment on cell A1. Step 5 writes a workbook whose first tab is Client Template with two data rows — Tesla and Citi — formatted exactly like the client sample.
About step 4: scripts/make_acceptance_decisions.py lives outside the product CLI on purpose. It fabricates an is_demo:false file so you can exercise the final path locally, but it stamps the reviewer as "SYNTHETIC ACCEPTANCE HELPER - not human reviewed" in every audit artifact. Real approvals always come from a person clicking through review.html. The product CLI has no way to manufacture human approvals.

The review gate — the heart of Phase 6

Decided by _evaluate_export_gate(). There are exactly three outcomes:

⛔ BLOCKED

Without --allow-unreviewed, any failing condition refuses the export with a non-zero exit and a list of reasons. Nothing is written.

📝 DRAFT

With --allow-unreviewed, the same problems become warnings; the file is written but visibly marked draft/unreviewed (tab name, colour, A1 comment, JSON flag).

✅ FINAL

A real (is_demo:false) decisions file with every required field resolved → a clean "Client Template" workbook, no draft markings.

What the gate refuses (without --allow-unreviewed)

Refuse when…Why it matters
decisions.is_demo == trueDemo decisions are auto-generated, not human review. They can only ever produce a DRAFT.
a required field has no decisionEvery one of the nine template columns must have been looked at.
attention field has no reviewer noteA flagged value needs an explicit, written human override — a bare approved isn't enough.
no metric extracted for a fieldNEW · hardening
The hardening (just added): the gate now walks all nine template columns per document. If extraction produced no metric at all for a required column, a FINAL export is refused with "No metric was extracted for required field …" instead of silently shipping a blank cell. Under --allow-unreviewed it becomes a warning. This is exactly the failure mode an unseen company (Phase 7) could trigger — now it fails loudly.

When does a cell actually get a value?

Passing the gate is necessary but not sufficient — _should_populate_cell() decides each cell independently:

Decision on the fieldResulting client cell
approved  (value present)populated
rejected / needs_fixblank
not_applicableblank
blank/null valueblank
So an attention field with needs_fix + note passes the gate (it was resolved) but its cell is left blank — the value didn't earn its way onto the client sheet. Only an explicit approved on a real value populates.

How raw numbers become client cells

map_metric_to_client_cell() converts each canonical value into the client's format — and refuses to guess if the units look wrong.

Template fieldStored asWritten asExample
Total revenue · Net income · Operating income · Operating expensesUSD / millionstext, billions (1 dp, trailing .0 stripped)22496 → $22.5B
Gross marginpercentage pointsdecimal, 0% format17.2 → 0.172 → "17%"
Earnings per shareUSD/sharenumber, "$"#,##0.000.33 → $0.33
Company · Quarter · Buybacks and dividendstexttext, verbatim"Tesla", "Q2 2025"
Defensive units: a currency cell is only written if unit=="USD" and scale=="millions"; gross margin only if unit=="percentage points". If the upstream normalizer ever changes and a value arrives in the wrong unit, the export raises rather than silently emitting a wrong $xB. Loud failure beats a quietly wrong client number.

Two documents → two rows, deterministically

🗂️ One row per source file

Metrics are grouped by their resolved source_file (robust even when a metric's company is missing), preserving first-seen draft order — so Tesla's row comes before Citi's, every time.

🎯 Duplicate tie-break

If two metrics claim the same column, one is chosen by a fixed priority: a cell-populating decision > a resolved-but-blanking decision > highest confidence > earliest in draft order. Ties are recorded as warnings in the JSON and audit report.

How Phase 6 is proven

tests/test_phase6_export.py locks the behaviour in — part of the 62 passing tests (was 61 before the absent-field hardening test). A sample of what's covered:

Mapping & formats

22496→$22.5B, 4100→$4.1B, 150000→$150B, 17.2→0.172, EPS stays numeric; bad currency units raise. Workbook headers/formats match the template exactly; all three audit tabs exist.

The gate

Demo blocked without override; draft with it; synthetic real decisions produce a true final; bare approved on an attention field is blocked; a non-attention rejected blanks the cell without a note; an absent template field blocks final but exports as draft with a warning.

Verified on the real data: Tesla eval 9/9, Citi eval 9/9, full suite 62 passed, ruff clean, project-map check clean, git diff --check clean. The no-cheat wall holds — export.py imports nothing from evaluation.
